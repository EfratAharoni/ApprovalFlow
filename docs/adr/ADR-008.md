# ADR-008: RAG over Policy — Vector Search Trade-offs (N5)

## הקשר

עד גרסה זו, סוכן ה-AI קיבל את `policy.md` כולו ב-system prompt בכל קריאה.
זה בזבזני מבחינת טוקנים ואינו מותאם לפריט הספציפי. כחלק מ-N5, המרנו
את `fetch_policy_clause` (כלי שדורש שה-LLM יידע מראש איזה rule_id לחפש)
לכלי `search_policy(query)` שמבצע חיפוש סמנטי על כל כללי הפוליסה ומחזיר
top-3 לפי cosine similarity.

בנוסף, לפני כל קריאת LLM, מבצעים חיפוש מקדים בעזרת `category + vendor + notes`
ומזריקים את 3 הסעיפים הרלוונטיים ביותר לפרומפט — במקום ה-policy המלא.

## ההחלטה

sentence-transformers `all-MiniLM-L6-v2` — מודל חינמי ומקומי (ללא API key),
~90 MB, רץ על CPU בביצועים סבירים. האינדקס נבנה פעם אחת ב-startup ומוחזק
בזיכרון. החיפוש מבוסס cosine similarity על embeddings מנורמלים.

## חלופות שנשקלו

**OpenAI text-embedding-3-small / Gemini embeddings — נדחו:**
דורשים API key נוסף, עלות לפי שימוש, ותלות ברשת בכל קריאה. לא מתאים
לדרישה "ללא שירותים בתשלום" ולרצון לשמור embedding מקומי.

**BM25 (חיפוש מילים מפתח) בלבד — נדחה:**
לא מטפל בשונות לשונית. query="overspending" לא יחזיר AUTONOMY-CEILING
שאומר "USD amount is ≤ $250". Vector search כן.

**BM25 + Vector (Hybrid) — נדחה לצורך הפרויקט הזה:**
שיפור ניכר, אבל דורש חישוב שני וריכוז ציונים (RRF/score fusion). מסובך
מדי לתחזוקה בהיקף הפרויקט הנוכחי. ראה "שיפורים אפשריים" למטה.

## כשלים ידועים — הגבלות ה-vector search

### ההבחנה החשובה: keyword overlap מול semantic gap אמיתי

**דוגמת "alcohol" — נראית כבעיה, אבל בעצם עובדת:**

```
query = "alcohol"
MEAL-03 text = "Alcohol-only receipts are not reimbursable."

→ cosine similarity גבוה כי "alcohol" מופיע ממש בטקסט.
→ זה keyword matching שנסתר מאחורי vector search — לא הוכחה ליכולת סמנטית.
```

**הכשל האמיתי — כאשר אין overlap לשוני בכלל:**

```
query = "team happy hour beverages reimbursement"
MEAL-03 text = "Alcohol-only receipts are not reimbursable."

→ "happy hour" / "beverages" לא חופפים ל-"alcohol-only" ברמת המילים.
→ המודל מדרג MEAL-01 ("team meals, $75/attendee") גבוה יותר כי "team" מופיע בשתיהן.
→ MEAL-03 לא מוחזר ב-top-3 — וה-LLM מחליט ללא הכלל הרלוונטי ביותר.
```

זה הכשל שמתועד ב-`test_known_gap_alcohol_paraphrase`.

### טבלת כשלים קונקרטיים

| Query בפועל | כלל נכון | כלל שמוחזר | למה |
|---|---|---|---|
| `"team happy hour beverages"` | MEAL-03 | MEAL-01 | "team" ≈ "team meals"; "beverages" לא ≈ "alcohol-only" |
| `"bar tab from client dinner"` | MEAL-03 | MEAL-02 | "client dinner" ≈ "client entertainment" (MEAL-02); "bar" ≈ לא ≈ "alcohol-only" |
| `"spending limit exceeded"` | AUTONOMY-CEILING | GLOBAL-FRAUD | "exceeded" + "limit" ≈ "fraud pattern" בחלל embedding כללי |
| `"missing paperwork"` | GLOBAL-RECEIPT | GLOBAL-MATH | "missing" ≈ "mismatch" יותר מ-"receipt not present" |

### למה זה קורה

all-MiniLM-L6-v2 מיומן על Common Crawl + Wikipedia, לא על שפת חשבונאות.
כשמשתמש כותב "bar tab" ופוליסה כותבת "alcohol-only", אין overlap לשוני
ואין מספיק training data שמחבר בין המושגים האלה בחלל ה-embedding.
המודל מוצא את השכן הקרוב ביותר — לא בהכרח את הרלוונטי ביותר.

## שיפורים אפשריים (אם היה זמן)

**1. Query Expansion:**
לפני החיפוש, להוסיף מילות מפתח מה-category לquery. לדוגמה:
```python
if event.category == "MEALS":
    query += " alcohol meal attendee entertainment"
```
פשוט, אפקטיבי, ניתן ליישום ב-30 דקות.

**2. Hybrid BM25 + Vector (Reciprocal Rank Fusion):**
לחשב ציון BM25 על טקסט הכלל במקביל לvector score, ולשקלל:
```
final_score = 0.5 * vector_score + 0.5 * bm25_score
```
מטפל בשונות לשונית (vector) ובהתאמת מילות מפתח (BM25) בו-זמנית.

**3. Fine-tuned Embedding Model:**
לאמן מחדש את המודל על זוגות (query, rule_id) מהיסטוריית הגשות.
דורש לפחות 500 דוגמאות, אבל נותן את השיפור הגדול ביותר.

**4. Reranking (Cross-Encoder):**
להריץ Cross-Encoder (כגון `cross-encoder/ms-marco-MiniLM-L-6-v2`) על
הtop-10 שהוחזרו כדי לדרג מחדש עם הקשר מלא. איטי יותר אבל מדויק הרבה יותר.

## תוצאות

**מה אנחנו מרוויחים:**
- חסכון בטוקנים: במקום ~500 טוקנים לpolicy מלא, מזריקים ~60 טוקנים (3 סעיפים)
- הcontex הרלוונטי ביותר מוזרק קודם → פחות "lost in the middle"
- LLM יכול לקרוא ל-search_policy ככלי אם הוא צריך סעיפים נוספים

**מה אנחנו מוותרים עליו:**
- סיכון ל-semantic drift בqueries לא פורמליים (מתועד ב-test_known_gap_alcohol_paraphrase)
- אם top-3 לא כולל את הכלל הנכון, הLLM עשוי להגיע להחלטה שגויה — אם כי
  הnull-safety של הrouter הדטרמיניסטי (router.py) מגן מפני רוב השגיאות האלו

# ADR-008: RAG over Policy — Vector Search Trade-offs (N5)

## Context

Until this version, the AI agent received the full `policy.md` in the system prompt on
every call. This is wasteful in tokens and not tailored to the specific item. As part of
N5, we replaced `fetch_policy_clause` (a tool that required the LLM to already know
which rule_id to look up) with `search_policy(query)` — a tool that performs semantic
search over all policy rules and returns the top-3 by cosine similarity.

Additionally, before each LLM call, we run a pre-search using `category + vendor + notes`
and inject only the 3 most relevant clauses into the prompt — instead of the full policy.

## Decision

sentence-transformers `all-MiniLM-L6-v2` — free, local model (no API key required),
~90 MB, runs on CPU with reasonable performance. The index is built once at startup and
held in memory. Search is based on cosine similarity over normalized embeddings.

## Alternatives Considered

**OpenAI text-embedding-3-small / Gemini embeddings — rejected:**
Require an additional API key, per-use cost, and network dependency on every call.
Incompatible with the "no paid services" requirement and the goal of keeping
embeddings local.

**BM25 (keyword search) only — rejected:**
Does not handle linguistic variation. `query="overspending"` would not return
AUTONOMY-CEILING which says "USD amount is ≤ $250". Vector search handles this.

**BM25 + Vector (Hybrid) — rejected for this project:**
A clear improvement, but requires a second computation and score fusion (RRF).
Too complex to maintain at this project's scope. See "Possible Improvements" below.

## Known Failures — Vector Search Limitations

### The Key Distinction: Keyword Overlap vs. True Semantic Gap

**The "alcohol" example — looks like a problem, but actually works:**

```
query = "alcohol"
MEAL-03 text = "Alcohol-only receipts are not reimbursable."

→ High cosine similarity because "alcohol" literally appears in the text.
→ This is keyword matching hidden behind vector search — not proof of semantic capability.
```

**The real failure — when there is zero lexical overlap:**

```
query = "team happy hour beverages reimbursement"
MEAL-03 text = "Alcohol-only receipts are not reimbursable."

→ "happy hour" / "beverages" do not overlap with "alcohol-only" at the word level.
→ The model ranks MEAL-01 ("team meals, $75/attendee") higher because "team" appears in both.
→ MEAL-03 is not returned in top-3 — and the LLM decides without the most relevant rule.
```

This is the failure documented in `test_known_gap_alcohol_paraphrase`.

### Concrete Failure Table

| Actual Query | Correct Rule | Rule Returned | Why |
|---|---|---|---|
| `"team happy hour beverages"` | MEAL-03 | MEAL-01 | "team" ≈ "team meals"; "beverages" ≉ "alcohol-only" |
| `"bar tab from client dinner"` | MEAL-03 | MEAL-02 | "client dinner" ≈ "client entertainment" (MEAL-02); "bar" ≉ "alcohol-only" |
| `"spending limit exceeded"` | AUTONOMY-CEILING | GLOBAL-FRAUD | "exceeded" + "limit" ≈ "fraud pattern" in general embedding space |
| `"missing paperwork"` | GLOBAL-RECEIPT | GLOBAL-MATH | "missing" ≈ "mismatch" more than "receipt not present" |

### Why This Happens

all-MiniLM-L6-v2 is trained on Common Crawl + Wikipedia, not on accounting/finance
language. When a user writes "bar tab" and the policy says "alcohol-only", there is no
lexical overlap and not enough training data connecting these concepts in embedding space.
The model finds the nearest neighbor — not necessarily the most relevant rule.

## Possible Improvements (Given More Time)

**1. Query Expansion:**
Before searching, add category-specific keywords to the query. For example:
```python
if event.category == "MEALS":
    query += " alcohol meal attendee entertainment"
```
Simple, effective, implementable in 30 minutes.

**2. Hybrid BM25 + Vector (Reciprocal Rank Fusion):**
Compute a BM25 score on rule text in parallel with the vector score, and blend:
```
final_score = 0.5 * vector_score + 0.5 * bm25_score
```
Handles linguistic variation (vector) and keyword matching (BM25) simultaneously.

**3. Fine-tuned Embedding Model:**
Retrain the model on (query, rule_id) pairs from submission history.
Requires at least 500 examples, but yields the largest improvement.

**4. Reranking (Cross-Encoder):**
Run a Cross-Encoder (e.g., `cross-encoder/ms-marco-MiniLM-L-6-v2`) over the
top-10 returned results to re-rank with full context. Slower but significantly
more accurate.

## Consequences

**What we gain:**
- Token savings: instead of ~500 tokens for the full policy, we inject ~60 tokens (3 clauses)
- The most relevant context is injected first → less "lost in the middle"
- The LLM can still call `search_policy` as a tool if it needs additional clauses

**What we give up:**
- Risk of semantic drift on informal queries (documented in `test_known_gap_alcohol_paraphrase`)
- If top-3 does not include the correct rule, the LLM may reach a wrong decision —
  though the null-safety of the deterministic router (router.py) protects against
  most of these errors
