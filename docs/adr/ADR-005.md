# ADR-005: Dapr State מול מסד נתונים לـ-HITL

## הקשר

דרישת M11 — תהליך הסלמה לאנושי חייב לשרוד restart של שירות בין הזמן
שהוא נעצר (paused) לזמן שהוא ממשיך (resume).

## ההחלטה

מצב הـ-workflow (`hitl:{submission_id}`) נשמר בـ-Dapr state store (Redis),
בנוסף לטבלת `approval_tasks` בـ-PostgreSQL שמשמשת כמקור האמת לתור עצמו.
בـ-startup, השירות סורק tasks עם `status=PENDING` בـ-DB ומשחזר את
הـ-state בـ-Redis אם הוא חסר.

## חלופות שנשקלו

**שמירת כל מצב הـ-workflow רק בـ-PostgreSQL, ללא Dapr state — נדחתה:**
Dapr state נותן API אחיד ופשוט (get/save/delete) שמופשט מהמימוש בפועל
(Redis כרגע, אבל ניתן להחלפה), ומאפשר atomic CAS (Compare-And-Swap) עם
ETags — שימושי גם עבור תקציבים (ראה INV-1014). שילוב של שני המאגרים
(Postgres כמקור אמת, Redis כקאש/state מהיר) נותן גם עמידות וגם ביצועים.

## תוצאות

**מה אנחנו מרוויחים:**
- שרידות מלאה לـ-restart
- הפשטה מהספק הספציפי (Dapr state יכול לרוץ מעל Redis, Postgres,
  Cosmos DB וכו')
- CAS אטומי למניעת race conditions

**מה אנחנו מוותרים עליו:**
- מורכבות נוספת — שני מקורות מידע (DB + Redis) שצריך לשמור מסונכרנים,
  ולוגיקת recovery בـ-startup שצריך לתחזק.

# ADR-005: Dapr State vs. Database for HITL

## Context

Requirement M11 — the human escalation flow must survive a service restart between
the time it is paused and the time it is resumed.

## Decision

The workflow state (`hitl:{submission_id}`) is stored in Dapr state store (Redis),
in addition to the `approval_tasks` table in PostgreSQL which serves as the source of
truth for the queue itself. On startup, the service scans tasks with `status=PENDING`
in the DB and restores state in Redis if it is missing.

## Alternatives Considered

**Storing all workflow state only in PostgreSQL, without Dapr state — rejected:**
Dapr state provides a unified, simple API (get/save/delete) abstracted from the actual
implementation (Redis currently, but swappable), and enables atomic CAS
(Compare-And-Swap) with ETags — also useful for budgets (see INV-1014). Combining
both stores (Postgres as source of truth, Redis as fast state/cache) provides both
durability and performance.

## Consequences

**What we gain:**
- Full resilience to restarts
- Abstraction from the specific backend (Dapr state can run on Redis, Postgres,
  Cosmos DB, etc.)
- Atomic CAS to prevent race conditions

**What we give up:**
- Additional complexity — two data sources (DB + Redis) that need to stay synchronized,
  and startup recovery logic that must be maintained.
