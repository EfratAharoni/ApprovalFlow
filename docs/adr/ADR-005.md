# ADR-005: Dapr State vs. DB for Durable HITL Pause

## Context
Requirement M11 — the human-escalation workflow must survive a service restart between the pause and the resume.

## Decision
Workflow state (`hitl:{submission_id}`) is stored in the Dapr state store (Redis), in addition to the `approval_tasks` table in PostgreSQL which serves as the authoritative source for the queue itself. On startup, the service scans for tasks with `status=PENDING` and restores any missing state in Redis.

## Alternatives Considered

**Storing all workflow state only in PostgreSQL — rejected.**
The Dapr state API provides a uniform interface (get/save/delete) that abstracts away the backing store, and supports atomic CAS (compare-and-swap) with ETags — useful for budget reservations (concurrent submissions to the same department).

## Consequences

**What we gain:**
- Full restart resilience
- Provider abstraction through Dapr
- Atomic CAS to prevent race conditions

**What we give up:**
- Two sources of truth (DB + Redis) must be kept in sync — handled by the recovery loop on startup and the timeout sweep every 10 minutes

# ADR-005: Dapr State מול DB ל-HITL Durable Pause

## הקשר
דרישת M11 — תהליך הסלמה לאנושי חייב לשרוד restart של שירות בין ה-pause לה-resume.

## ההחלטה
מצב ה-workflow (`hitl:{submission_id}`) נשמר ב-Dapr state store (Redis), בנוסף לטבלת `approval_tasks` ב-PostgreSQL שמשמשת כמקור האמת לתור עצמו. ב-startup, השירות סורק tasks עם `status=PENDING` ומשחזר state חסר ב-Redis.

## חלופות שנשקלו

**שמירת כל מצב ה-workflow רק ב-PostgreSQL — נדחתה.**
Dapr state נותן API אחיד (get/save/delete) שמופשט מהמימוש בפועל, ומאפשר CAS אטומי עם ETags — שימושי גם עבור תקציבים (submissions מקבילים לאותה מחלקה).

## תוצאות

**מה מרוויחים:**
- שרידות מלאה ל-restart
- הפשטה מהספק הספציפי דרך Dapr
- CAS אטומי למניעת race conditions

**מה מוותרים עליו:**
- שני מקורות מידע (DB + Redis) שצריך לשמור מסונכרנים — מטופל על ידי recovery loop ב-startup ו-timeout sweep כל 10 דקות
