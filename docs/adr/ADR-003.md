# ADR-003: תקרת אוטונומיה — $250 וכללים מבוססי קטגוריה

## הקשר

הפוליסה של Northwind Components (policy.md §6) מגדירה ברירת מחדל של
$250 לפריט כתקרת אישור אוטומטי. עלינו להחליט האם לאמץ את ברירת
המחדל, להעלות אותה, או להוריד אותה — ולהצדיק את הבחירה בכתב
בקובץ docs/PRODUCT-DILEMMA.md כנדרש על ידי הפוליסה.

הדילמה: תקרה גבוהה מדי חושפת את הארגון לסיכון פיננסי אם הסוכן
טועה. תקרה נמוכה מדי מרוקנת את האוטומציה מערך ומעמיסה על
מאשרים בבקשות טריוויאליות.

## ההחלטה

אנחנו מאמצים את ברירת המחדל של הפוליסה: $250 לפריט.

פריט מאושר אוטומטית אם ורק אם כל התנאים הבאים מתקיימים יחד:

| תנאי | ערך |
|---|---|
| סכום (USD לאחר המרה) | ≤ $250 |
| ציון ביטחון הסוכן | ≥ 0.80 |
| עמידה בכללי הקטגוריה | ✓ (MEAL-01, SAAS-01, HW-01, TRAVEL-01) |
| אין hard-stop פעיל | ✓ |

Hard-stops שמסלימים תמיד — ללא קשר לסכום או ביטחון:

| rule_id | מצב |
|---|---|
| GLOBAL-VENDOR | ספק חדש/לא מוכר |
| GLOBAL-RECEIPT | חסר קבלה על פריט מעל $25 |
| GLOBAL-FX | פריט מטבע זר מעל $1,000 לאחר המרה |
| GLOBAL-MATH | סכום שורות + מס ≠ total |
| GLOBAL-FRAUD | אותת הונאה |
| MEAL-01 | ארוחת צוות ללא ספירת משתתפים |
| MEAL-02 | בידור לקוח מעל $500 ללא הצדקה עסקית |
| MEAL-03 | קבלה של אלכוהול בלבד → reject |
| TRAVEL-02 | נסיעה בודדת מעל $1,500 |
| TRAVEL-03 | כרטיס ראשונה/עסקי |
| HW-02 | חומרה מעל $1,000 |

ארבעת ה-routes של הראוטר הדטרמיניסטי:
- `auto_approve`  — כל התנאים מתקיימים
- `human_review`  — חריגה מתקרה, hard-stop, או ביטחון נמוך
- `reject`        — הפרה חמורה (MEAL-03)
- `duplicate`     — GLOBAL-DUP: אותו vendor + invoiceNumber + total

הסוכן רק ממליץ. הראוטר הדטרמיניסטי בקוד הוא שמחליט את ה-route
הסופי ואוכף את התקרה — בלי תלות בפלט ה-LLM.

## חלופות שנשקלו

**$500 — נדחתה:**
ניתוח sample-invoices.json מראה שאין פריט בטווח $250–$500 שעובר
את כל השערים ממילא. INV-1018 הוא $220 אבל נכשל ב-SAAS-01,
ו-INV-1013 הוא $300 עם הערת "Approve me" ונכשל בתקרה.
העלאה ל-$500 לא תוסיף אישורים אוטומטיים אמיתיים ותיצור
חשיפה לא מוצדקת בשלב ההשקה.

**$100 — נדחתה:**
תורידה כיסוי אוטומטי מתחת לרף משמעותי. פריטים כמו
INV-1002 ($99) ו-INV-1016 ($48) יעברו בשתי ההגדרות —
אין יתרון אבטחתי ממשי בהורדה.

## תוצאות

**מה אנחנו מרוויחים:**
- עמידה בברירת המחדל של הפוליסה
- לפחות 4 פריטים מאושרים אוטומטית: INV-1001, INV-1002, INV-1016, INV-1017
- רדיוס פגיעה מקסימלי מוגבל ל-$250
- מסר ברור: המערכת שמרנית בשלב ההשקה

**מה אנחנו מוותרים עליו:**
- פריטים לגיטימיים בטווח $250–$500 יגיעו לתור האנושי.
  בפועל כל הפריטים בטווח זה נכשלים בשער אחר ממילא.
- נשקול העלאה ל-$350 אחרי 90 יום תפעול תקין.

# ADR-003: Autonomy Ceiling — $250 and Category-Based Rules

## Context

Northwind Components' policy (policy.md §6) defines a default of $250 per item as the
autonomous approval threshold. We needed to decide whether to adopt this default, raise it,
or lower it — and justify the choice in writing in docs/PRODUCT-DILEMMA.md as required
by the policy.

The dilemma: a ceiling that is too high exposes the organization to financial risk if the
agent makes a mistake. A ceiling that is too low strips the automation of value and
overloads approvers with trivial requests.

## Decision

We adopt the policy default: $250 per item.

An item is auto-approved if and only if all of the following conditions are met simultaneously:

| Condition | Value |
|---|---|
| Amount (USD after conversion) | ≤ $250 |
| Agent confidence score | ≥ 0.80 |
| Compliance with category rules | ✓ (MEAL-01, SAAS-01, HW-01, TRAVEL-01) |
| No active hard-stop | ✓ |

Hard-stops that always escalate — regardless of amount or confidence:

| rule_id | Condition |
|---|---|
| GLOBAL-VENDOR | New/unknown vendor |
| GLOBAL-RECEIPT | Missing receipt for item over $25 |
| GLOBAL-FX | Foreign-currency item over $1,000 after conversion |
| GLOBAL-MATH | Line items + tax ≠ total |
| GLOBAL-FRAUD | Fraud signal detected |
| MEAL-01 | Team meal without attendee count |
| MEAL-02 | Client entertainment over $500 without business justification |
| MEAL-03 | Alcohol-only receipt → reject |
| TRAVEL-02 | Single trip over $1,500 |
| TRAVEL-03 | First/business class ticket |
| HW-02 | Hardware over $1,000 |

The four routes of the deterministic router:
- `auto_approve`  — all conditions met
- `human_review`  — ceiling breach, hard-stop, or low confidence
- `reject`        — severe violation (MEAL-03)
- `duplicate`     — GLOBAL-DUP: same vendor + invoiceNumber + total

The agent only recommends. The deterministic router in code makes the final routing
decision and enforces the ceiling — with no dependency on LLM output.

## Alternatives Considered

**$500 — rejected:**
Analysis of sample-invoices.json shows no item in the $250–$500 range that passes all
gates anyway. INV-1018 is $220 but fails SAAS-01, and INV-1013 is $300 with a note
saying "Approve me" and fails the ceiling. Raising to $500 would add no real automatic
approvals and would create unjustified exposure at launch.

**$100 — rejected:**
Would lower automatic coverage below a meaningful threshold. Items like INV-1002 ($99)
and INV-1016 ($48) pass under both settings — there is no real security benefit to
lowering the limit.

## Consequences

**What we gain:**
- Compliance with the policy default
- At least 4 items auto-approved: INV-1001, INV-1002, INV-1016, INV-1017
- Maximum blast radius capped at $250
- Clear message: the system is conservative at launch

**What we give up:**
- Legitimate items in the $250–$500 range will go to the human queue.
  In practice every item in that range fails another gate anyway.
- We will consider raising to $350 after 90 days of normal operation.
