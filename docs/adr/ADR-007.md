# ADR-007: API Gateway — Nginx vs. Custom Proxy

## Context
Requirement M6 — a single external entry point with rate-limiting. A technology choice was required.

## Decision
Nginx with the built-in `limit_req` module.

## Alternatives Considered

**Custom gateway in Python/FastAPI — rejected.**
Writing a custom gateway requires manual implementation of rate-limiting, routing, and failure handling — all of which are already solved and battle-tested in Nginx. Our needs (static routing + rate-limiting) are simple enough that Nginx is the right tool.

**Managed gateway services (Azure API Management, AWS API Gateway) — rejected.**
These require a dependency on a specific cloud provider and exceed the "no paid services" constraint.

## Consequences

**What we gain:**
- Battle-tested stability
- High throughput
- Simple single-file configuration
- No cloud dependency

**What we give up:**
- Programmatic flexibility — dynamic routing logic in the future would require migrating to a code-based gateway (e.g. Traefik, or a custom FastAPI proxy)

# ADR-007: API Gateway — Nginx מול Proxy מותאם אישית

## הקשר
דרישת M6 — נקודת כניסה חיצונית יחידה עם rate-limiting. צריך להחליט על הטכנולוגיה.

## ההחלטה
Nginx עם מודול `limit_req` מובנה.

## חלופות שנשקלו

**Gateway מותאם אישית ב-Python/FastAPI — נדחתה.**
כתיבת gateway עצמאי דורשת מימוש ידני של rate-limiting, routing וטיפול בכשלים — כל אלה כבר פתורים ובדוקים-בקרב ב-Nginx. הצרכים שלנו (ניתוב סטטי + rate-limiting) פשוטים מספיק ש-Nginx הוא הפתרון המתאים.

**שירותי Gateway מנוהלים (Azure API Management) — נדחו.**
דורשים תלות בענן ספציפי וחורגים מהדרישה "ללא שירותים בתשלום."

## תוצאות

**מה מרוויחים:**
- יציבות מוכנה בקרב
- ביצועים גבוהים
- קונפיגורציה פשוטה בקובץ אחד
- אין תלות בענן

**מה מוותרים עליו:**
- גמישות תכנותית — לוגיקת ניתוב דינמית בעתיד תדרוש מעבר ל-gateway מתוכנת
