# ADR-006: טופולוגיית מסד הנתונים — Postgres משותף עם סכמות נפרדות

## הקשר

לכל מיקרו-שירות אמורה להיות בעלות בלעדית על הנתונים שלו (עיקרון יסוד
מ-MSA, שיעור 6 ו-8). היה צריך להחליט בין מופע Postgres נפרד לכל שירות
לבין מופע משותף עם הפרדה לוגית.

## ההחלטה

מופע PostgreSQL אחד, עם סכמה (schema) נפרדת לכל שירות:
`submission`, `decisions`, `approvals`, `payments`, `audit`.
כל שירות מתחבר רק לסכמה שלו, ואין אף שירות שניגש ישירות לסכמה
של שירות אחר.

## חלופות שנשקלו

**מופע Postgres נפרד לכל שירות (5 קונטיינרים) — נדחתה לצורך הפרויקט הזה:**
זו הגישה ה"נכונה" יותר מבחינה ארכיטקטונית טהורה (בידוד מלא, סקיילינג
עצמאי), אבל מוסיפה תקורת זיכרון ותחזוקה משמעותית (5 קונטיינרי DB,
5 connection pools, 5 backup strategies) שלא הצדיקה את עצמה בהיקף
ובזמן של הפרויקט. בפועל, הפרדת הסכמות נותנת את אותה בעלות לוגית
ומונעת גישה ישירה בין שירותים — בלי לשלם את המחיר התפעולי המלא.

## תוצאות

**מה אנחנו מרוויחים:**
- בעלות לוגית מלאה (כל שירות עם migrations משלו, אף שירות לא ניגש
  לטבלה של שירות אחר)
- חיסכון משמעותי במשאבים בסביבת dev/לימוד
- פשטות בהרצה עם docker compose

**מה אנחנו מוותרים עליו:**
- בידוד פיזי מלא — תקלה במופע הـ-Postgres עצמו (לא בסכמה) משפיעה
  על כל השירותים יחד. ברור שזה trade-off מודע: בפרודקשן אמיתי בקנה
  מידה גדול, הייתי ממליצה לעבור למופעים נפרדים.

# ADR-006: Database Topology — Shared Postgres with Separate Schemas

## Context

Each microservice should have exclusive ownership of its own data (a core MSA principle
from Lessons 6 and 8). We needed to decide between a dedicated Postgres instance per
service versus a shared instance with logical separation.

## Decision

A single PostgreSQL instance with a separate schema per service:
`submission`, `decisions`, `approvals`, `payments`, `audit`.
Each service connects only to its own schema, and no service directly accesses another
service's schema.

## Alternatives Considered

**A separate Postgres instance per service (5 containers) — rejected for this project:**
This is the "more correct" approach architecturally (full isolation, independent scaling),
but adds significant memory and operational overhead (5 DB containers, 5 connection pools,
5 backup strategies) that was not justified at this project's scope and timeline.
In practice, schema separation provides the same logical ownership and prevents
cross-service direct access — without paying the full operational cost.

## Consequences

**What we gain:**
- Full logical ownership (each service has its own migrations, no service accesses
  another service's tables)
- Significant resource savings in dev/learning environment
- Simplicity when running with docker compose

**What we give up:**
- Full physical isolation — a failure in the Postgres instance itself (not in the schema)
  affects all services together. This is a deliberate trade-off: in a real production
  environment at scale, I would recommend migrating to separate instances.
