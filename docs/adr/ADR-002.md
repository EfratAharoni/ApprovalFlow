# ADR-002: Choreography vs. Orchestration for the Saga

## Context
The payment flow (M9) requires coordination across multiple services with compensation on failure. The decision was whether to use a central orchestrator (Orchestration) or have each service react to events independently (Choreography).

## Decision
Choreography — each service listens to pub/sub events and responds autonomously, with no central coordinator.

## Alternatives Considered

**Orchestration with a central saga manager — rejected.**
Advantage: easier to follow the full workflow from one place.
Disadvantage: creates a Single Point of Failure and a potential bottleneck, and conflicts with the independence principle of microservices. The project uses Dapr pub/sub as infrastructure — Choreography fits naturally.

## Consequences

**What we gain:**
- No central SPOF
- Each service is fully autonomous
- High fault tolerance

**What we give up:**
- Harder to trace the complete flow end-to-end
- Mitigated by `correlation_id` and the audit-service, which centralises all events by `submission_id`

# ADR-002: Choreography מול Orchestration ל-Saga

## הקשר
תהליך התשלום (M9) דורש קואורדינציה בין כמה שירותים עם פיצוי בעת כשל. צריך להחליט האם יש מתזמן מרכזי (Orchestration) או שכל שירות מגיב לאירועים בעצמו (Choreography).

## ההחלטה
Choreography — כל שירות מאזין לאירועי pub/sub ומגיב באופן עצמאי, ללא רכיב מרכזי.

## חלופות שנשקלו

**Orchestration עם saga manager מרכזי — נדחתה.**
יתרון: קל יותר לעקוב אחרי הזרימה ממקום אחד.
חסרון: יוצר Single Point of Failure ו-bottleneck פוטנציאלי, ומנוגד לעיקרון העצמאות של מיקרו-שירותים. הפרויקט משתמש ב-Dapr pub/sub כתשתית — Choreography מתאים טבעית.

## תוצאות

**מה מרוויחים:**
- אין SPOF מרכזי
- כל שירות עצמאי לחלוטין
- עמידות גבוהה לכשלים

**מה מוותרים עליו:**
- קשה יותר לעקוב אחרי הזרימה הכוללת
- מקוזז על ידי `correlation_id` ו-audit-service שמרכז את כל האירועים לפי `submission_id`
