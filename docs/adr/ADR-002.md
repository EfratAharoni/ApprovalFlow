# ADR-002: Choreography מול Orchestration לـ-Saga

## הקשר

תהליך התשלום (M9) דורש קואורדינציה בין כמה שירותים עם פיצוי בעת כשל.
צריך להחליט האם יש מתזמן מרכזי שמנהל את התהליך (Orchestration) או שכל
שירות מגיב לאירועים בעצמו (Choreography).

## ההחלטה

Choreography — כל שירות מאזין לאירועי pub/sub ומגיב באופן עצמאי,
ללא רכיב מרכזי שמנהל את הـ-saga.

## חלופות שנשקלו

**Orchestration עם saga manager מרכזי — נדחתה:**
יתרון: קל יותר לעקוב אחרי הזרימה ממקום אחד. אבל יוצר נקודת כשל
בודדת (Single Point of Failure) ו-bottleneck פוטנציאלי, וזה מנוגד
לעיקרון המרכזי של מיקרו-שירותים — "הגישה האנוכית של השירות" (שיעור 6).
בנוסף, פרויקט זה משתמש בـ-Dapr pub/sub כתשתית טבעית לـ-choreography ממילא.

## תוצאות

**מה אנחנו מרוויחים:**
- אין SPOF
- כל שירות עצמאי לחלוטין ויכול להיפרס/להתעדכן בנפרד
- עמידות גבוהה יותר לכשלים

**מה אנחנו מוותרים עליו:**
- קשה יותר לעקוב אחרי הזרימה הכוללת — מקטינים את הבעיה הזו עם
  `correlation_id` ו-audit-service שמרכז את כל האירועים לפי
  `submission_id`, כך שעדיין יש "תצוגה אחת" של הזרימה גם בלי
  מתזמן מרכזי.

# ADR-002: Choreography vs. Orchestration for the Saga

## Context

The payment flow (M9) requires coordination between several services with compensation
on failure. We needed to decide whether a central coordinator manages the process
(Orchestration) or each service reacts to events independently (Choreography).

## Decision

Choreography — each service listens to pub/sub events and reacts independently,
with no central component managing the saga.

## Alternatives Considered

**Orchestration with a central saga manager — rejected:**
Advantage: easier to trace the full flow from one place. However, it creates a Single
Point of Failure and a potential bottleneck, which contradicts a core microservices
principle — "the selfish service approach" (Lesson 6). Additionally, this project already
uses Dapr pub/sub as a natural infrastructure for choreography.

## Consequences

**What we gain:**
- No SPOF
- Each service is fully independent and can be deployed/updated separately
- Higher resilience to failures

**What we give up:**
- Harder to trace the overall flow — we mitigate this with `correlation_id` and the
  audit-service that centralizes all events by `submission_id`, so there is still a
  "single view" of the flow even without a central coordinator.
