# ADR-009 — JWT Authentication at the API Gateway

## Context / הקשר

Any caller could approve or reject any submission — there was no identity check. Before the
system could handle real money, at least the approval/rejection endpoints needed to be gated
to authorised reviewers.

Two placement options exist:

| Option | Where verification happens |
|---|---|
| **A — Gateway** | Nginx `auth_request` calls an auth sidecar; services receive only pre-verified requests |
| **B — Each service** | Every service independently validates the `Authorization` header |

## Decision / החלטה

**Option A — verification at the Nginx API Gateway** via `auth_request`.

A dedicated `auth-service` (FastAPI, port 8006) issues JWTs (`POST /auth/token`) and verifies
them (`GET /auth/verify`). Nginx calls `/auth/verify` as an internal subrequest before
forwarding protected requests; a 401/403 from the auth service is returned directly to the
caller without the upstream ever seeing the request.

Protected endpoints (approver or admin role required):
- `POST /approvals/{id}/decide`

All other endpoints remain open (submitters, audit consumers, monitoring).

## Alternatives Considered / חלופות שנשקלו

| Alternative | Reason rejected |
|---|---|
| Middleware in each service | Duplicates verification logic across 6 services; a missed update leaves one service unguarded |
| API-key header (no JWT) | No role information; can't distinguish approver from submitter |
| OAuth2 / external IdP | No paid/external services constraint; adds infra complexity out of scope |
| mTLS between services | Solves service-to-service auth, not end-user identity |

## Consequences / השלכות

- Services remain stateless and role-unaware — they process whatever reaches them
- Role enforcement is centralised: one place to add, remove, or audit access rules
- Auth service is a new dependency for the API gateway; if it is unreachable, protected routes
  return 503 (nginx upstream failure) — acceptable for a single-region deployment
- `JWT_SECRET_KEY` must be kept in `.env` (not committed); rotating it invalidates all active tokens
- `auth-service` added to `docker-compose.yml`, CI quality-gates, and the GHCR publish job

## עברית

אימות JWT מתבצע בשכבת ה-Nginx בלבד — השירותים עצמם נשארים stateless וללא מודעות לתפקידים.
`auth-service` נפרד מנפיק טוקנים (`POST /auth/token`) ומאמת אותם (`GET /auth/verify`).
Nginx קורא ל-`/auth/verify` כ-subrequest פנימי לפני העברת כל בקשה מוגנת.
