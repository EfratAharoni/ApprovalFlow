# ADR-004: Idempotency Strategy

## Context
Requirement M10 — the same submission, a redelivered event, or a retried payment must all produce exactly one effect.

## Decision
Server-side idempotency key computed as:
`SHA256(vendor + amount_cents + invoiceNumber + date)`

This key is checked against the database before creating a new submission. In addition: `UNIQUE(submission_id)` on the payments table, and `UNIQUE(event_id)` on the audit_events table prevent duplicate processing at every layer.

## Alternatives Considered

**Client-supplied key (Idempotency-Key header, as in the Stripe API) — rejected.**
Requires the client to generate and persist a unique key. The server-side hash approach places no behavioural requirements on the caller at all.

## Consequences

**What we gain:**
- Client cannot accidentally break idempotency
- All logic is centralised on the server
- Easy to test and verify

**What we give up:**
- Two submissions with identical fields (same vendor, same amount, same invoice number, same date) will be treated as a duplicate — an edge case we accepted as low-risk

# ADR-004: אסטרטגיית Idempotency

## הקשר
דרישת M10 — אותה הגשה, אותו אירוע שנמסר פעמיים, או אותה קריאת תשלום — חייבים להניב תוצאה זהה.

## ההחלטה
מפתח אידמפוטנטיות מחושב בצד השרת:
`SHA256(vendor + amount_cents + invoiceNumber + date)`

נבדק מול מסד הנתונים לפני יצירת submission חדש. בנוסף: `UNIQUE(submission_id)` בטבלת payments, ו-`UNIQUE(event_id)` בטבלת audit_events.

## חלופות שנשקלו

**מפתח מסופק על ידי הלקוח (Idempotency-Key header, כמו ב-Stripe API) — נדחתה.**
דורשת תיאום מצד הלקוח לייצר ולשמור מפתח ייחודי. גישת ה-hash בצד השרת לא דורשת שום שינוי התנהגות מצד הלקוח.

## תוצאות

**מה מרוויחים:**
- לקוח לא יכול לשבור אידמפוטנטיות בטעות
- הלוגיקה מרוכזת בשרת
- קל לבדוק

**מה מוותרים עליו:**
- שני submissions שונים עם אותם נתונים בדיוק יייחשבו כפילות — תרחיש קצה נדיר שקיבלנו עליו את הסיכון
