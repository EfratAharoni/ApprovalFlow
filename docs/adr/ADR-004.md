# ADR-004: אסטרטגיית Idempotency

## הקשר

דרישת M10 — אותה הגשה, אותו אירוע שנמסר פעמיים, או אותה קריאת תשלום
— חייבים להניב תוצאה זהה תמיד.

## ההחלטה

מפתח אידמפוטנטיות מחושב בצד השרת: `SHA256(vendor + amount_cents +
invoiceNumber + date)`. נבדק מול Dapr state לפני יצירת submission חדש.
בנוסף — `UNIQUE(submission_id)` בטבלת payments, ו-`UNIQUE(event_id)`
בטבלת audit_events.

## חלופות שנשקלו

**מפתח אידמפוטנטיות שמסופק על ידי הלקוח (idempotency-key header,
כמו ב-Stripe API) — נדחתה:**
דורשת תיאום מצד הלקוח (UI/API consumer) לייצר ולשמור מפתח ייחודי,
מה שמוסיף מורכבות לצד הצרכן. גישת הـ-hash בצד השרת לא דורשת שום
שינוי התנהגות מצד הלקוח — היא נגזרת אוטומטית מתוכן ההגשה.

## תוצאות

**מה אנחנו מרוויחים:**
- לקוח לא יכול "לשבור" אידמפוטנטיות בטעות
- הלוגיקה מרוכזת בשרת
- קל לבדוק

**מה אנחנו מוותרים עליו:**
- שני submissions שונים באמת עם אותם נתונים בדיוק (vendor + amount +
  invoiceNumber + date זהים) ייחשבו כפילות גם אם הכוונה הייתה אחרת
  — תרחיש קצה נדיר מאוד שקיבלנו עליו את הסיכון.

# ADR-004: Idempotency Strategy

## Context

Requirement M10 — the same submission, the same event delivered twice, or the same
payment call — must always produce the same result.

## Decision

A server-side computed idempotency key: `SHA256(vendor + amount_cents + invoiceNumber + date)`.
Checked against Dapr state before creating a new submission. Additionally —
`UNIQUE(submission_id)` on the payments table, and `UNIQUE(event_id)` on audit_events.

## Alternatives Considered

**Client-supplied idempotency key (idempotency-key header, like the Stripe API) — rejected:**
Requires the client (UI/API consumer) to generate and store a unique key, adding complexity
on the consumer side. The server-side hash approach requires no behavioral change from
the client — it is derived automatically from the submission's content.

## Consequences

**What we gain:**
- The client cannot accidentally "break" idempotency
- Logic is centralized on the server
- Easy to test

**What we give up:**
- Two genuinely different submissions with identical data (same vendor + amount +
  invoiceNumber + date) will be treated as duplicates even if intentional — a very rare
  edge case whose risk we accept.
